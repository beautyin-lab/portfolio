# PRD: 웹 에이전시 포트폴리오 사이트 + 30개 업종별 데모 사이트

**문서 버전**: 1.0
**작성일**: 2026-03-18
**작성자**: Senior Project Manager

---

## 1. 프로젝트 개요 및 목표

### 1.1 프로젝트 요약

웹 에이전시의 역량을 보여주는 포트폴리오 메인 사이트와, 15개 업종 카테고리별 2개씩 총 30개의 실제 동작하는 데모 사이트를 구축한다. 잠재 고객이 자신의 업종에 맞는 데모를 직접 체험하고, 바로 문의로 이어지는 영업 파이프라인을 만드는 것이 핵심 목표이다.

### 1.2 목표

| 구분 | 목표 | 측정 기준 |
|------|------|-----------|
| 비즈니스 | 문의 전환율 향상 | 데모 체험 후 문의 폼 제출률 5% 이상 |
| 비즈니스 | 신뢰도 확보 | 30개 업종별 실동작 데모로 "우리 업종도 가능하다"는 확신 제공 |
| 기술 | 빠른 로딩 | 모든 페이지 Lighthouse Performance 90점 이상 |
| 기술 | 유지보수 효율 | Turborepo 모노레포로 공통 컴포넌트 재사용, 단일 배포 파이프라인 |
| 품질 | 크로스 디바이스 | 375px ~ 1440px 4개 브레이크포인트에서 레이아웃 깨짐 0건 |

### 1.3 프로젝트 범위

**포함 (In Scope)**:
- 메인 포트폴리오 사이트 1개
- 업종별 데모 사이트 30개 (15 카테고리 x 2)
- 문의 폼 데이터 D1 저장 및 관리
- 데모 사이트 MockBackend (실제 서버 없이 동작하는 시뮬레이션)
- 반응형 디자인 (375 / 768 / 1024 / 1440px)

**미포함 (Out of Scope)**:
- 실제 결제 연동
- 회원가입/로그인 시스템
- 관리자 대시보드 (CMS)
- 실제 예약/주문 백엔드 (MockBackend로 대체)
- 다국어 지원

## 2. 타겟 사용자

### 2.1 주요 타겟

**소규모 사업자 (Primary)**
- 연령: 30~50대
- 예산: 300만원대
- 특성: 웹사이트의 필요성은 인지하나, 어떤 형태가 좋은지 판단이 어려움
- 니즈: "내 업종에 맞는 사이트가 실제로 어떻게 보이는지" 미리 확인하고 싶음
- 디바이스: 모바일 60%, 데스크톱 40% (초기 검색은 모바일, 상세 확인은 데스크톱)

**예비 창업자 (Secondary)**
- 사업 준비 단계에서 웹사이트 제작 업체를 미리 탐색
- 가격 대비 퀄리티를 비교하는 경향이 강함

### 2.2 사용자 시나리오

1. 소규모 카페 사장님이 "카페 홈페이지 제작"을 검색하여 메인 사이트에 도달
2. 카테고리 갤러리에서 "카페/베이커리" 카테고리를 선택
3. 2개의 데모 사이트를 직접 체험 (메뉴 보기, 매장 위치 확인 등)
4. 가격 안내 페이지에서 300만원대 패키지 확인
5. 문의 폼을 통해 견적 요청

## 3. 핵심 기능 목록

### 3.1 메인 포트폴리오 사이트

#### 3.1.1 랜딩 페이지 (Hero)
- 에이전시 핵심 메시지와 CTA (문의하기 / 데모 보기)
- 풍부한 인트로 애니메이션 (스크롤 트리거, 페이드인, 패럴랙스)
- 대표 데모 사이트 3~4개 미리보기 카드

#### 3.1.2 포트폴리오 갤러리
- 15개 카테고리 필터 (탭 또는 태그 방식)
- 카테고리 선택 시 해당 업종 데모 2개를 카드로 표시
- 카드 클릭 시 데모 사이트로 이동 (새 탭 또는 라우팅)
- 카드 호버 애니메이션 (스케일업, 그림자, 미리보기 전환)

#### 3.1.3 가격 안내
- 3단계 패키지 (Basic / Standard / Premium) 비교 테이블
- 각 패키지 포함 항목 명시
- "가장 인기" 배지로 Standard 패키지 강조

#### 3.1.4 문의 폼
- 입력 필드: 이름, 연락처, 이메일, 업종 (드롭다운), 예산 범위, 요청 사항 (텍스트에어리어)
- 클라이언트 사이드 유효성 검사
- Cloudflare D1에 저장
- 제출 성공/실패 피드백 UI

#### 3.1.5 FAQ
- 아코디언 UI
- 최소 10개 항목 (제작 기간, 수정 횟수, 유지보수, 호스팅 등)

#### 3.1.6 고객 후기
- 카드 슬라이더 (캐러셀)
- 업종, 고객명(이니셜), 별점, 후기 텍스트
- 목업 데이터 최소 6건

#### 3.1.7 공통 요소
- 반응형 네비게이션 (데스크톱: 수평 메뉴, 모바일: 햄버거)
- 부드러운 스크롤 네비게이션 (앵커 링크)
- 푸터 (연락처, SNS 링크, 저작권)
- 페이지 전환 및 스크롤 기반 애니메이션

### 3.2 데모 사이트 (30개)

#### 3.2.1 공통 구조
모든 데모 사이트는 다음 공통 요소를 포함한다:

- 업종에 맞는 히어로 섹션
- 서비스/상품 소개 섹션
- 오시는 길 / 위치 정보 (정적 지도 이미지 또는 임베드)
- 연락처 / 문의 섹션
- 반응형 레이아웃 (375 / 768 / 1024 / 1440px)
- "이 사이트는 데모입니다" 플로팅 배너 + 포트폴리오 메인으로 돌아가기 링크

#### 3.2.2 디자인 아키타입 (10종)

30개 데모 사이트는 아래 10개 디자인 아키타입 중 하나를 적용하여 시각적 차별화를 달성한다.

| # | 아키타입 | 특징 | 적합 업종 예시 |
|---|----------|------|----------------|
| 1 | **클린 미니멀** | 여백 중심, 산세리프, 모노톤 | 법률, 부동산 |
| 2 | **웜 내추럴** | 따뜻한 색감, 둥근 모서리, 부드러운 그라데이션 | 카페, 베이커리, 꽃집 |
| 3 | **볼드 다이나믹** | 강렬한 컬러, 큰 타이포, 대각선 레이아웃 | 헬스장, 피트니스 |
| 4 | **엘레강트 럭셔리** | 다크 배경, 골드 악센트, 세리프 폰트 | 웨딩, 뷰티 |
| 5 | **프렌들리 플레이풀** | 밝은 파스텔, 일러스트풍 아이콘, 라운드 카드 | 펫, 키즈 |
| 6 | **프로페셔널 트러스트** | 블루/그레이 톤, 정돈된 그리드, 인증 배지 | 의료, 학원/교육 |
| 7 | **포레스트 힐링** | 그린 계열, 자연 이미지, 부드러운 애니메이션 | 웰니스, 펜션/숙박 |
| 8 | **모던 인더스트리얼** | 그레이/블랙, 직선적 레이아웃, 대비 강한 타이포 | 인테리어, 스터디카페 |
| 9 | **비비드 팝** | 선명한 다색, 카드 기반, 사진 중심 | 식당/맛집 |
| 10 | **소프트 그래디언트** | 그래디언트 배경, 글래스모피즘, 라운드 UI | 범용 (변주용) |

#### 3.2.3 업종별 MockBackend 기능

MockBackend은 실제 서버 통신 없이 클라이언트 사이드에서 동작하는 시뮬레이션이다. 데이터는 JSON 파일 또는 인메모리로 관리한다.

| 15 | **식당/맛집** | 메뉴판(사진 포함), 영업시간, 오시는 길, 예약, 후기 | 메뉴 카테고리 탭, 예약 폼 (날짜, 시간, 인원, 좌석 유형 선택) | 비비드 팝 | 웜 내추럴 |

### 5.1 공통 MockBackend 동작 원칙

1. **데이터 소스**: 각 데모별 `/data/[demo-slug].json`에 목업 데이터를 정적 파일로 관리
2. **폼 제출**: 실제 전송 없이 2초 로딩 후 성공 모달 표시, 입력값은 콘솔에 로깅
3. **필터/검색**: 클라이언트 사이드 JSON 필터링으로 즉각 반응
4. **캘린더/좌석**: 랜덤 시드 기반으로 날짜/좌석별 가용 상태를 생성하여 실제감 부여
5. **가격 계산**: 선택 항목에 따른 합계를 클라이언트에서 실시간 계산

## 8. 제약 사항 및 가정

### 8.1 제약 사항

| 구분 | 내용 |
|------|------|
| 이미지 소스 | Unsplash 또는 Picsum만 사용. Pexels 사용 금지 (403 오류 발생) |
| 백엔드 | 실제 백엔드 서버 없음. 모든 동적 기능은 MockBackend (클라이언트 사이드) |
| 데이터베이스 | D1은 문의 폼 저장 용도로만 사용. 데모 데이터는 정적 JSON |
| 배포 환경 | Cloudflare Workers 런타임 제한 (CPU 시간 10ms free / 50ms paid, 메모리 128MB) |
| 외부 API | 지도는 정적 이미지 또는 iframe 임베드. 별도 API 키 필요 서비스는 사용하지 않음 |
| 프로세스 | 백그라운드 프로세스 실행 금지. 서버 시작 명령 불포함 (개발 서버는 이미 실행 중이라고 가정) |

### 8.2 가정

1. 디자인 시안 없이 개발자가 디자인 아키타입 기반으로 직접 구현한다
2. 콘텐츠(텍스트)는 목업 데이터를 사용하며, 실제 고객 데이터가 아니다
3. 도메인 및 DNS 설정은 별도로 진행한다 (이 프로젝트 범위 밖)
4. 개발 환경에 Node.js 20+, pnpm이 설치되어 있다
5. Cloudflare 계정 및 Workers/D1 서비스가 사전에 설정되어 있다
6. 한국어 단일 언어로 제작한다
7. 첫 번째 배포 후 2~3회 수정 사이클이 있을 수 있다

### 8.3 리스크 및 완화 방안

| 리스크 | 영향 | 완화 방안 |
|--------|------|-----------|
| 30개 데모의 디자인 일관성 부족 | 퀄리티 불균형 | 10개 아키타입 기반 디자인 토큰으로 일관성 확보 |
| 이미지 로딩 지연 | Lighthouse 점수 하락 | Next.js Image 최적화, 적절한 사이즈 지정, lazy loading |
| MockBackend 복잡도 증가 | 개발 일정 지연 | 공통 유틸리티 패키지화, 단순 시뮬레이션에 집중 |
| CF Workers 런타임 제한 초과 | 배포 실패 | RSC 활용 최소화, 정적 생성(SSG) 우선, 에지 함수 경량화 |
| 브레이크포인트별 레이아웃 이슈 | QA 통과 실패 | Tailwind 반응형 유틸리티 일관 사용, Playwright 자동화 테스트 |
